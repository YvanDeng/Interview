## 一、Packed Pointer vs Tagged Pointer vs Non-pointer

这三个是 64 位时代苹果用来"榨干地址空间"的设计，层层递进。

### 1. Raw Pointer（原始指针）

最朴素的理解：

```
┌────────────────────────────────────────┐
│ 0x0000 1005 7c3a 2840                  │  ← 64位地址，直接指向对象
└────────────────────────────────────────┘
```

地址的低位始终为 0（因为 64 位系统按 8 字节对齐，地址总是 8 的倍数），
所以实际上地址的**低 3 位永远用不上。



对于内存地址 `0x0000 1005 7c3a 2840`，我们可以从两个层面来理解其高低地址：

**从整个地址值来看**

这就像看一个十进制数 `123456` 一样，左边的数字代表更大的数量级。

- **高地址部分：** `0x0000 1005`
- **低地址部分：** `0x7c3a 2840`

在 Objective-C 中，**对象本身**存储在**堆区**，而这个地址值是一个**指针**，它存储的内存位置（通常**在栈上**）就保存着这个地址值。

**从计算机内部存储来看 (字节序)**

当这个地址值被存入内存时，它会被拆分成一个个字节。这些字节在内存中如何排列，取决于计算机的**字节序 (Endianness)**。目前主流的 iOS 设备（ARM架构）和模拟器（x86_64架构）在默认情况下都使用**小端序 (Little-Endian)**。

**小端序的规则是：数据的低位字节存储在内存的低地址处。**

我们将地址 `0x0000 1005 7c3a 2840` 拆分为单个字节，从高位到低位依次是：`00`, `00`, `10`, `05`, `7c`, `3a`, `28`, `40`。

在**小端序**的内存中，它们的存储顺序如下：

| 内存地址 (从低到高) | 存储的字节值 | 对应原地址的部分     |
| ------------------- | ------------ | -------------------- |
| **低地址** →        | `0x40`       | **最低位字节 (LSB)** |
| ↓                   | `0x28`       |                      |
| ↓                   | `0x3A`       |                      |
| ↓                   | `0x7C`       |                      |
| ↓                   | `0x05`       |                      |
| ↓                   | `0x10`       |                      |
| ↓                   | `0x00`       |                      |
| **高地址** →        | `0x00`       | **最高位字节 (MSB)** |

所以，总结来说：

- 从地址值的书写习惯看，左边是高地址部分，右边是低地址部分。
- 从计算机内存的实际存储看（小端序），地址值的低位字节 (`0x40`) 存放在内存的低地址，高位字节 (`0x00`) 存放在内存的高地址。
- **1 个字节 = 8 位二进制，而 1 位十六进制数刚好能代表 4 位二进制。所以，8 除以 4 等于 2，正好需要两位十六进制数**



### 2. Non-pointer ISA（未打包的纯指针 ISA）

就是 raw pointer，isa 存的是 class 对象的地址，64 位全部用来表示地址。

**问题**：64 位地址空间根本用不完（苹果只映射了 42 位虚拟地址），大量 bit 是"死"的。

### 3. Packed ISA（打包指针 ISA）—— arm64 的核心优化

利用地址中"永远是 0 的那些 bit"，把 metadata 塞进去：

```
┌────────────────────┬──┬──┬───────┐
│ class pointer (33b)  │01│W │extra_rc│
└────────────────────┴──┴──┴───────┘
 ↑ 上部33位存class地址  ↑ ↑  ↑ isa内引用计数
   nonpointer=1      weak rc
```

**控制位（从低位到高位）**：

| 位 | 名称 | 含义 |
|----|------|------|
| 0 | **nonpointer** | 1 = packed isa；0 = raw isa（legacy/64-bit clean） |
| 1 | **weakly_referenced** | 是否有 weak 指针指向此对象 |
| 2 | **deallocating** | 对象是否正在 dealloc |
| 3.. | **extra_rc** | isa 内嵌的低位引用计数 |

**为什么是 33 位？** 苹果的虚拟地址空间只映射了 33 位（2^33 = 8GB 用户空间），所以第 33 位以上的 bit 全部是 0，这些空闲 bit 全拿来做控制位。

### 4. Tagged Pointer（Tagged Pointer ISA）——更进一步

如果说 Packed ISA 是"把 metadata 塞进地址空位"，那 **Tagged Pointer 是彻底放弃指针语义**，直接用指针本身存数据：

```
┌───────┬──────────────┬──────┐
│ 1 (T) │   payload     │ tag  │
└───────┴──────────────┴──────┘
 ↑ MSB=1   数据内容          类型
```

**触发条件**：苹果用 isa 地址的**最高位**（MSB）= 1 来标记 tagged pointer

**特征**：
- 不再指向任何真实对象，`object_getClass` 返回 nil
- 指针本身是数据，没有 isa、没有引用计数
- 无法用 `retain`/`release` 操作——直接修改指针值

**典型应用**：

```objc
NSNumber *n = @42;         // tagged pointer，地址里存 42
NSString *s = @"hello";   // tagged pointer（小字符串）

// 判断
if ((uintptr_t)s & (1UL << 63)) {
    // 是 tagged pointer
}

// 解码 tag 和 payload
// tag = 低 4 位 或 特定 bits
// payload = 剩余 bits
```

### 三者对比

| | Raw Pointer | Packed ISA | Tagged Pointer |
|---|---|---|---|
| **isa** | 64-bit 纯地址 | 64-bit 地址 + 控制位 | MSB=1，isa=nil |
| **控制位** | 无 | nonpointer/weak/extra_rc | tag + payload |
| **对象？** | 是 | 是（真实对象） | 伪对象（指针=数据） |
| **retain/release** | ✅ 正常 | ✅ 正常 | ❌ 无效 |
| **出现场景** | armv7/老代码 | **arm64 所有对象** | 短字符串、小数字 |

### 关键面试回答

> "Packed ISA 和 Tagged Pointer 都是 64 位的优化，但目的不同。"
>
> - **Packed ISA**：解决"地址用不完但需要存 metadata"的问题，把引用计数、weak 标记塞进 isa 的空闲 bit，**对象还是真实存在的**。
> - **Tagged Pointer**：更进一步，**彻底放弃指针语义**，直接用指针的 MSB 做 tag，payload 存数据，完全不需要分配内存，常见于 `NSNumber` 和短 `NSString`。
> - 判断方式：`(uintptr_t)obj & (1UL << 63)`，MSB=1 即 tagged pointer。

## 二、内存地址的方向（高地址 vs 低地址）

### 标准约定

```
高地址  0xFFFFFFFFFFFF  ←─────── 增长方向 ───────→  0x000000000000  低地址
        栈 (stack)                                堆 (heap)
         ↓ 生长                                   ↑ 生长
```

**常见困惑："向上"是涨还是落？**

| 说法 | 描述 |
|------|------|
| 地址值增大 | `0x1000` → `0x2000` → `0x3000`（往"上"走） |
| 栈往下生长 | `SP -= 16`（地址**减小**） |
| 堆往上生长 | `ptr = malloc(16)` → 返回**更大**的地址 |

所以**"栈向低地址生长，堆向高地址生长"**，两者对着长，最大化利用虚拟地址空间。

### 对象内存布局图（含方向标注）

**一个有两个 CGFloat ivar 的对象（arm64）：**

```
          高地址
┌─────────────────────────────┐  ← 0x...F008
│         isa (8B)            │
├─────────────────────────────┤
│       ivar_1 (8B)           │  CGFloat
├─────────────────────────────┤
│       ivar_2 (8B)           │  CGFloat
├─────────────────────────────┤
│      padding (0~7B)         │  对齐到 16B
└─────────────────────────────┘  ← 0x...F000  低地址

class_getInstanceSize() = 32B
malloc_size()          = 32B (nano zone) 或 48B (size class)
```

**malloc header（不在对象内部，在对象低地址外侧）：**

```
                                    高地址
┌──────────────────────────────────────┐
│  malloc_header_t {                  │
│    size_t size;       (8B)          │
│    size_t prev_size;  (8B)          │  ← 用于合并相邻空闲块
│    uintptr_t next;   (8B)           │
│    uintptr_t prev;   (8B)          │
│    uint32_t allocated;(4B)          │
│    ... padding ...                  │
│  }                                 │
├──────────────────────────────────────┤
│                                      │
│          对象本身                    │  ← 用户可见区域
│    (isa + ivars + padding)          │
│                                      │
└──────────────────────────────────────┘
                                      低地址
```

**关键口诀**：

> **"栈沉低，堆冒高"**
> - 栈：向低地址走（`SP = SP - 帧大小`）
> - 堆：向高地址走（`brk`/`mmap` 返回值越来越大）

### 常见面试坑

**Q：对象内的 ivar 排列顺序是低地址到高地址还是反过来？**

```
低地址  ─────────────────────  高地址
┌────────┬────────┬─────────────┐
│  isa   │ ivar_1 │   ivar_2   │
└────────┴────────┴─────────────┘
```

**答**：从 isa 开始，往高地址排。第一个 ivar 的地址 = isa 地址 + 8B。越后声明的 ivar 地址越高。

**验证代码**：
```objc
typedef struct {
    void *isa;
    CGFloat a;  // +8
    CGFloat b;  // +16
    CGFloat c;  // +24
} Obj;

CGFloat *p = (CGFloat *)((char *)obj + 8);
NSLog(@"a=%p, b=%p, c=%p", p, p+1, p+2);
// p+1 比 p 大 8（因为 sizeof(CGFloat)=8）
```

## 三、打印的对象地址是高地址还是低地址？

**打印的地址 = 对象内存布局的最低地址 = isa 所在的位置。**

```objc
NSObject *obj = [[NSObject alloc] init];
NSLog(@"%p", obj);  // 打印的就是 isa 的地址，也是这块内存的起始地址
```

这是 C 语义：`&obj` 就是取对象**首字节**的地址，不管对象占多少字节。

所以：
- `0x1000` 打印出来是 `0x1000`
- 这个对象的 ivar 们分布在 `0x1000 + 8`、`0x1000 + 16` ...
- malloc header 在 `0x1000 - 16`（对象外，在更低地址处）

**一句话**：地址值最小的那个位置。

## 四、Q1 原始答案修正

原答案中关于 isa 后面 bits 的描述不够准确，修正如下：

**arm64 isa bits 结构（`union isa_t`）**：

```
┌──────────────────────────┬─────┬─────┬───────────┐
│  class pointer (upper 33b) │  1  │  W  │ extra_rc  │
│  (取自 isa & 0x1fffffff800) │ np  │    │           │
└──────────────────────────┴─────┴─────┴───────────┘
                                 1bit 1bit   ≥19bits
```

- **nonpointer**：arm64 下始终为 1，表示这是 packed isa
- **weakly_referenced**：是否有 weak 指针指向此对象
- **extra_rc**：isa 内嵌的低位引用计数，高位引用计数存在 SideTable

**引用计数不**在对象头部单独占字段——存在 isa 的 extra_rc 或 Side Table 中。

**最小对象（没有任何 ivar 的 NSObject）**：

```
┌─────────────────────────────────────┐
│  isa (8B) — packed pointer + bits  │  ← 起始地址 = 0x...
├─────────────────────────────────────┤
│  最小分配 16B（即使没有任何 ivar） │
└─────────────────────────────────────┘
```
