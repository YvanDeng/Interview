## 一、内存布局与结构

### Q1: 一个 Objective-C 对象在内存中长什么样？请描述它的基本结构。
<details>
<summary>参考答案（点击展开）</summary>

**核心结构（arm64）：**

```
┌─────────────────────┬──────────────────────┬─────────────────────┐
│     isa (8B)        │   弱引用标记 (1B)    │   引用计数 (4B/8B)  │
├─────────────────────┴──────────────────────┴─────────────────────┤
│                        实例变量区 (N × 8B)                        │
└──────────────────────────────────────────────────────────────────┘
```

- **isa**：指向 Class 对象（或 Meta-Class）的指针，arm64 下是 64 位（8 字节）
- **弱引用标记 + 引用计数**：联合存储在 isa 的额外 bits 中（`has_swi_Table`、`weakly_referenced`、`unretained_pointee_owner` 等）
- **实例变量**：按 8 字节对齐，编译器按声明顺序或优化顺序排列

**最小尺寸**：16 字节（即使没有任何实例变量，对象也至少占 16 字节）
</details>

---

### Q2: 对象头的 isa 指针占多少字节？为什么 arm64 架构下是 64 位？
<details>
<summary>参考答案（点击展开）</summary>

- **arm64**：isa 指针为 64 位（8 字节），因为地址总线宽度为 64 位，可寻址空间为 2^64
- **armv7**：isa 为 32 位（4 字节）

**注意**：64 位不等于用了 64 位物理内存，内存地址实际只用到其中一部分（苹果实际只映射了 42 位虚拟地址空间）。

**Tagged Pointer 特殊处理**：如果启用了 Tagged Pointer，isa 的最后一位被用作标记，真正的数据压缩存储在指针本身中，不再指向任何对象。
</details>

---

### Q3: 你知道 Class 对象和 Meta-Class 对象吗？它们分别放在内存的哪个区域？它们的关系是怎样的？
<details>
<summary>参考答案（点击展开）</summary>

**内存区域**：都存储在堆（heap）和元数据区（non-pointer isa 区域），但不属于堆。

```
┌──────────────┐     ┌──────────────┐
│  Class对象   │────→│ Meta-Class   │
│ (实例方法)   │     │ (类方法)     │
└──────┬───────┘     └──────┬───────┘
       │                   │
       └───────┬───────────┘
               ↓
        ┌──────────────┐
        │  Root Meta   │ → 指向自己（形成闭环）
        └──────────────┘
```

**关系**：
- `Object` 的 isa 指向 `Class`
- `Class` 的 isa 指向 `Meta-Class`
- `Meta-Class` 的 isa 指向 `Root Meta-Class`（通常是 `NSObject` 的 meta-class）
- 类方法存在 Meta-Class 中，因为类方法本质是"发送给类对象的消息"

**重要特性**：
- Category 添加的方法会合并到主 Class（或 Meta-Class）中
- 方法查找时，消息转发链会依次搜索：类方法缓存 → 类方法列表 → Meta-Class → ... → Root Meta-Class
</details>

### Q4: 如果我打印一个类的 `class_getInstanceSize` 和 `malloc_size`，这两个值一定相等吗？为什么？
<details>
<summary>参考答案（点击展开）</summary>

**不一定相等，且通常 `malloc_size > class_getInstanceSize`。**

**原因**：
- `class_getInstanceSize`：返回"对象本身所需字节数"（对齐到 8 字节）
- `malloc_size`：返回"实际分配的字节数"（由内存分配器决定）

**malloc 分配规则（Nano allocator）**：
- 小对象（< 256B）：按 16/32/48/.../256 字节的 size class 分配
- 大对象（≥ 256B）：按页（4KB）对齐分配

**例如**：一个只有 2 个 CGFloat 成员的对象（16B），实际会被分配 32B（nano zone）或 48B（libsystem 的 size class）。

所以：`malloc_size` >= `class_getInstanceSize` 恒成立。
</details>

## 二、ARC 与引用计数

### Q5: retain 和 release 到底做了什么？能描述一下 Side Table 的结构吗？
<details>
<summary>参考答案（点击展开）</summary>

**retain 流程：**
1. 找到对象的 isa（或 extra isa）
2. 检查是否启用 Side Table（引用计数过大时）
3. 若在 isa 中：用 `RC_WORD`（一个 bitmap）原子增加计数
4. 若在 Side Table：`sidetable_retain()` 原子操作 Side Table 中的引用计数

**release 流程：**
1. 找到对象的 isa
2. 从 RC_WORD 或 Side Table 原子减一
3. 若结果为 0：触发 dealloc 流程

**Side Table 结构（`SideTable`）：**
```objc
struct SideTable {
    spinlock_t slock;           // 自旋锁，保证原子性
    RefcountMap refcnts;        // 是一个 map<DisguisedPtr<objc_object>, uintptr_t>
    weak_table_t weak_table;    // 弱引用表
};
```

**引用计数存储方式（二选一）：**
- **inline（isa 内）**：isa 的 extra bits 中存引用计数（最多  RCOBJECT_INLINE_RETAIN_COUNT_MAX = 4095）
- **Side Table**：超出上限后移到 Side Table 的 refcnts map 中

**为什么用 map 而不是数组？** 因为对象的创建是动态的，用全局 map（以对象地址为 key）可以避免预先分配巨大数组，且便于按需淘汰。
</details>

---

### Q6: 引用计数是怎么存储的？如果一个对象引用计数接近 INT_MAX，会发生什么？
<details>
<summary>参考答案（点击展开）</summary>

**存储方式（三种）：**

1. **isa extra bits（inline）**：arm64 下 isa 指针有优化的 bits 可存储引用计数（小对象）
2. **Side Table refcnts**：大引用计数或有 weak 引用时使用
3. **Side Table weak_table**（独立存储 weak 指针本身，不是引用计数）

**溢出问题**：
- Side Table 用 `uintptr_t` 存储引用计数，理论上可到 2^64-1
- 但 ARC 语义下，正常程序很难触达极限
- 真正危险的是**悬垂指针**（dangling pointer）和**重复释放**，而不是计数溢出
- ⚠️ `NSInteger` 本身可能在某些计算中溢出（尤其涉及 `CFGetRetinCount()` 返回 `CFIndex` 时）

**思考题**：如果引用计数从 4095 继续 retain，会发生什么？（触发 Side Table 迁移，之后计数移到堆上的 map）
</details>

### Q7: weak 指针是如何做到在对象销毁后自动置 nil 的？weak_table 是怎么组织的？
<details>
<summary>参考答案（点击展开）</summary>

**dealloc 时 weak clear 流程：**

```
dealloc
  → objc_object::clearDeallocating()
  → sidetable_clearWeak()
  → weak_table_t[alias] → 找到所有指向该对象的 weak 指针地址
  → 逐一置 nil（*weak_ptr = nil）
  → 从 weak_table 中删除该条目
```

**weak_table 结构：**
```objc
struct weak_table_t {
    weak_entry_t *entries;     // hash map，key = 对象地址
    size_t    num_entries;     // 条目数
    uintptr_t mask;            // hash mask
    uintptr_t max_hash_displacement; // 防碰撞上限
};

struct weak_entry_t {
    weak_referrer_t referrers[WEAK_INLINE_COUNT]; // 固定条目数组
    // 超出时动态分配 DisguisedPtr<objc_object*> referrers[];
    // 存储所有指向 this 对象地址的 weak 变量指针地址
};
```

**hash 插入**：以对象的 `DisguisedPtr`（地址取反）为 key，插入 referrer（weak 指针的地址）的地址

**为什么用 referrer 而非 referent？** 因为 weak 指针是"值"（存储在某对象的某个 ivar 槽位中），我们需要通过对象地址找到所有"指向它的 weak 变量"，以便清理。

**为什么对象销毁时能找到所有 weak 指针？** 因为 weak 指针**不会**存储对象的地址，而是存储自己（变量本身）的地址在 weak_table 中。
</details>

### Q8: 当一个对象收到 release 消息后，判断是否要 free 的依据是什么？Dealloc 的完整流程是什么？
<details>
<summary>参考答案（点击展开）</summary>

**判断条件**：
- **inline RC**：检查 isa extra bits 中的引用计数是否归零
- **Side Table**：检查 refcnts map 中是否归零
- **合并判断**：`!has_swi_Table && RC_ONE(extra_rc) == 0` 则触发 dealloc

**Dealloc 完整流程：**

```
release
  → objc_object::release()
  → 对象引用计数 -1
  → 判断 RC 是否为 0
  → objc_object::dealloc()
  → _objc_rootDealloc(self)
  → rootDealloc()
    ┌─────────────────────────┐
    │ 若有 C++ 析构函数       │→ ~Object() 调用
    │ 若有 associated objects │→ object_removeAssocations()
    │ 引用计数 > 0？          │（Side Table cleanup）
    │ 是 weak 引用表？        │→ weak_table_clearNoLock()
    │ 不是 Category 的 dealloc？│（跳过，直接到最后）
    └─────────────────────────┘
  → free(obj)  // 归还给 malloc heap
```

**注意**：ARC 下 `dealloc` 不需要手动调用，但可以重写 `dealloc` 做一些清理（如 KVO 移除、通知中心移除观察者）。

**关键问题**：如果对象 A 引用了对象 B，B 的 dealloc 会先执行还是 A 的？（答案：子对象先 dealloc）
</details>

## 三、Autorelease Pool

### Q9: 调用 NSLog(@"%@", obj) 时，如果 obj 是 autorelease 对象，它什么时候被释放？
<details>
<summary>参考答案（点击展开）</summary>

**NSLog 会持有对象**：格式化时会通过 `NSStringFromClass`、`description` 等获取对象的字符串表示，这个过程会产生 autorelease。

**释放时机取决于线程的 autorelease pool 何时 pop**：

```
┌────────────────────────────┐
│  @autoreleasepool {        │
│      id obj = [NSString...autorelease];  // 标记 A
│      NSLog(@"%@", obj);    │  ← 此时 obj 还在 pool 中
│  }  // ← pop 时释放标记 A  │
└────────────────────────────┘
│  但如果是：                 │
│  id obj = [str autorelease];
│  dispatch_async(^{         │  ← 线程切换，pool 已 pop
│      NSLog(@"%@", obj);   │  ← 可能已被释放（EXC_BAD_ACCESS）
│  });
└────────────────────────────┘
```

**在主线程 runloop 的每一次迭代中**，Cocoa 会自动在 RunLoop 事件前 push pool，事件后 pop，所以主线程通常不会因为 autorelease 泄漏。

**但**：UI 事件处理中如果 autorelease 对象过多，可能在下一次 drain 前导致内存峰值积压。
</details>

### Q10: @autoreleasepool {} 到底在做什么？它和线程本地存储（TLS）有什么关系？
<details>
<summary>参考答案（点击展开）</summary>

**底层实现（简化）：**
```objc
// 编译器转换
@autoreleasepool {
    // 被转换成：
    void *ctx = objc_autoreleasePoolPush();
    // ... 用户代码 ...
    objc_autoreleasePoolPop(ctx);
}
```

**TLS（Thread Local Storage）关系**：
- 每个线程都有一个**隐式的 autorelease pool page 链表
- 通过 TLS 键 `NSAutoreleasePoolKey`（`__PTK`) 访问当前线程的 pool 栈
- `push()`：在当前线程 TLS 中压入新 page，返回哨兵对象地址
- `pop(sentinel)`：弹出到哨兵位置，逐一 release 该 page 内所有对象

**结构图**：
```
线程 TLS
  │
  ▼
autorelease_page_t *currentPage ──→ [obj, obj, obj, ..., sentinel]
                                 ──→ [obj, obj, ..., sentinel]
                                 ──→ ...
```

**为什么用 TLS 而不是全局队列？** 因为 autorelease 是线程独立的，全局队列会有锁竞争，TLS 避免加锁，实现零开销的线程隔离。
</details>

### Q11: main 函数的第一行就用了 @autoreleasepool {}，这是为什么？如果去掉会怎样？
<details>
<summary>参考答案（点击展开）</summary>

**main.m 典型代码：**
```objc
int main(int argc, char * argv[]) {
    @autoreleasepool {
        return UIApplicationMain(...);
    }
}
```

**原因**：
- `UIApplicationMain` 内部会触发大量 autorelease 对象（Class 加载、Bundle 解析、Storyboard/XIB 加载等）
- 如果没有最外层的 pool，这些对象会在**主线程 RunLoop 第一次 dormir 之前**一直驻留内存
- 可能导致**启动阶段内存峰值过高**，甚至触发 Memory Warning

**去掉会怎样？**
- 不会立即崩溃（RunLoop 会自动管理 pool）
- 但启动期间所有 autorelease 对象会等到第一次 `[NSRunLoop drain]` 才释放
- 在 iOS 低内存设备上可能触发 Jetton Diesel 或 early memory pressure

**结论**：最佳实践——**尽早 push**，**尽量小的 scope**，让 autorelease 对象尽快释放。
</details>

## 四、Retain Cycle 与排查

### Q12: 为什么 NSTimer 常常是内存泄漏的元凶？请从数据结构角度分析。
<details>
<summary>参考答案（点击展开）</summary>

**典型场景：**
```objc
@interface MyVC ()
@property (nonatomic, strong) NSTimer *timer;
@end

@implementation MyVC
- (void)startTimer {
    self.timer = [NSTimer scheduledTimerWithTimeInterval:1.0
        target:self
        selector:@selector(tick)
        userInfo:nil
        repeats:YES];
    // Timer retain 了 self（target）
    // self.timer retain 了 timer
    // 形成 cycle：self ↔ timer
}
@end
```

**Timer 的数据结构 retain 链：**
```
NSTimer (in timerList)
  └── target: self (MyVC)
        └── timer property → NSTimer
              ↑ 互相持有
```

**原因**：
- RunLoop 持有 timer（timerList 是 runloop 的成员）
- Timer 持有 target（`target:self`）
- VC 持有 timer（`self.timer`）
- **Cycle**：VC ↔ Timer

**解决方案**：
1. `invalidate` + `self.timer = nil`（在 `viewWillDisappear` 或 `dealloc` 中）
2. 封装一个 `TimerTarget` 拦截器，用 weak self 打破循环
3. GCD `dispatch_source_t` timer（无 target retain 问题）

**数据结构角度**：Timer 内部用 `target` 存储回调目标，这是一个**强引用的闭包**，是 cycle 的根源。
</details>

### Q13: Block 捕获外部对象时，什么情况下会形成 retain cycle？怎么用 clang -rewrite-objc 验证？
<details>
<summary>参考答案（点击展开）</summary>

**形成 cycle 的条件**：Block 被某对象持有，同时 Block 内部捕获了该对象，形成 `self → block → captured_vars → self`。

**典型例子：**
```objc
self.block = ^{
    [self doSomething]; // 捕获 self（strong by default）
};
// self.block → block → self（cycle）
```

**clang -rewrite-objc 验证**：
```bash
clang -rewrite-objc -fobjc-arc MyFile.m
```
会生成 `.cpp` 文件。可以看到 Block 的结构：
```cpp
struct __MyVC__doSomething_block_impl_0 {
    struct __block_impl impl;
    struct __MyVC__doSomething_block_desc_0* Desc;
    MyVC *__strong self; // 捕获方式
};
```

**ARC 下 Block 捕获规则：**
- 捕获**基本类型**：值传递（不产生 retain）
- 捕获**对象类型**：默认 `__strong`，引用计数 +1
- 捕获**__block 修饰的变量**：在 Block 内部可修改，但可能造成循环引用（`__block x = self` → block → self）
- 捕获**__weak 变量**：不增加引用计数，用于打破循环

**最佳实践**：
```objc
__weak typeof(self) weakSelf = self;
self.block = ^{
    __strong typeof(weakSelf) strongSelf = weakSelf;
    if (strongSelf) [strongSelf doSomething];
};
```
</details>

### Q14: 给你一个疑似泄漏的界面，如何快速定位泄漏的根源对象？你会使用哪些工具，步骤是什么？
<details>
<summary>参考答案（点击展开）</summary>

**工具组合**：
1. **Instruments / Leaks** — 动态监测
2. **Instruments / Allocations** — 记录对象生命周期
3. **Memory Graph Debugger（Xcode）** — 瞬时抓取 retain cycle
4. **Malloc Stack Logging** — 追踪分配堆栈

**排查步骤**：

```
Step 1: 用 Instruments Allocations 记录一次完整交互
        → Mark Heap（进入界面）
        → 触发所有操作
        → Mark Heap（离开界面）
        → 两次 Mark 之间的差异对象即为泄漏候选

Step 2: Memory Graph Debugger
        → 在泄漏点暂停
        → 找到泄漏对象的引用链（who retains me）
        → 顺着链找到 cycle 闭环

Step 3: 如果找不到，看 malloc stack
        → 开启 Malloc Stack（Zombies 或 Allocations with stack）
        → 找到对象分配时的调用栈
        → 定位是哪个模块创建了这个对象

Step 4: 验证
        → 尝试在 dealloc 加 log，或重写 dealloc 确认是否真的没释放
```

**常见泄漏场景 TOP 5**：
1. NSTimer / CADisplayLink 未 invalidate
2. Block 循环引用
3. delegate / dataSource 未设 nil
4. NotificationCenter 观察者未移除
5. 复杂 VC 层级中的强引用链（Parent → Child → Parent）

## 五、内存分配算法

### Q15: malloc 分配一个对象时，内存布局从低地址到高地址依次是什么？
<details>
<summary>参考答案（点击展开）</summary>

**对象内存布局（从低到高）：**

```
低地址
┌──────────────┬──────────────┬───────────────┬──────────────┐
│ 对象头 (16B) │  isa + bits  │  实例变量     │ 对齐填充     │
├──────────────┴──────────────┴───────────────┴──────────────┤
│              malloc 头部 (sizeof(malloc_header_t))          │
│              → size + flags + prev/next 指针               │
├───────────────────────────────────────────────────────────┤
│              用户可见的内存（上面 isa + ivars）            │
│                                                         │
└───────────────────────────────────────────────────────────┘
                                                          高地址
```

**malloc tiny vs malloc nano：**
- **nano zone**（< 16B 且对齐）：直接用 zone 的 bitmap 管理，无 malloc_header
- **nano zone**（16~256B）：有最小 16B 粒度的 slab 分配
- **size class zone**（> 256B）：使用 4KB 页对齐，malloc_header 在对象前方

**malloc_header_t 结构**（简化）：
```c
struct malloc_header_t {
    size_t size;              // 当前块大小（含 header）
    size_t previous_size;     // 前一个块大小（用于合并）
    struct malloc_header_t *next;  // 空闲链表 next
    struct malloc_header_t *prev;  // 空闲链表 prev
    uint32_t allocated;       // 标记：0=free, 1=allocated
};
```

### Q16: 你了解苹果的 os_allocator 和 malloc_zone_t 吗？它的分配算法有什么优势？
<details>
<summary>参考答案（点击展开）</summary>

**os_allocator**（libmalloc）：
- 苹果的内存分配器（Darwin/MacOS/iOS），不是 `glibc malloc`
- 支持 **nano zone**（小对象优化）和 **magzone**（大对象按需映射）

**malloc_zone_t 是内存区域（zone）的抽象接口**：
```c
typedef struct {
    malloc_zone_t *global_zone;  // 默认 zone
    // 每个线程可以有自己的 zone（thread-local malloc）
} malloc_zone_t;

struct malloc_zone_ops {
    void *(*malloc)(malloc_zone_t *, size_t);
    void *(*realloc)(malloc_zone_t *, void *, size_t);
    void  (*free)(malloc_zone_t *, void *);
    // ... 还有牙 teporary / force_lock 等操作
};
```

**分配算法**：
1. **Segratated Heap（分离式堆）**：小对象（≤ 15KB）用 Size Class 的 Free List；大对象直接 mmap
2. **Bitmap 扫描**：nano zone 用位图而不是空闲链表，减少碎片
3. **madvise(MADV_FREE)**：释放内存时不立即归还物理页，标记为可复用，减少系统调用开销

**优势**：
- 零碎片：Size class 将分配大小离散化，减少内部碎片
- 锁优化：热点加锁（自旋锁），大多数操作无锁
- mmap 按需映射：大对象内存不会长期占用物理页

### Q17: 什么是内存碎片？苹果用哪些策略来对抗碎片化？
<details>
<summary>参考答案（点击展开）</summary>

**内存碎片类型**：
- **内部碎片**：分配的内存块比实际需求大（size class 粒度造成）
- **外部碎片**：空闲内存总和足够，但无法满足大块分配请求（空闲块太小且分散）

**苹果的对抗策略**：

**1. Size Classes（分离空闲链表）**
```
Zone 中维护多个 free list，按 size class 分组：
free_list[16]  → 16B 空闲块
free_list[32]  → 32B 空闲块
free_list[48]  → 48B 空闲块
...
free_list[256] → 256B 空闲块
```
分配时找最近匹配的 class，减少内部碎片。

**2. 合并（Coalescing）**
- 释放时，检查相邻块是否也空闲
- `previous_size` + 当前块向前合并
- `current_size` + 下一个块向后合并
- 形成更大的连续空闲块 → 减少外部碎片

**3. Bitmap 扫描（nano zone）**
- 每个 slab（page）用 4 个 word 的 bitmap 追踪分配状态
- 空闲查找从低位往高位扫描 O(1) 找到第一个 0
- 无需维护链表头，metadata 更少

**4. MADV_FREE 策略**
- 释放时 `madvise(addr, size, MADV_FREE)`
- 物理页立即可用（按需换出），但虚拟地址保留
- 下次分配时直接复用，无需向内核重新申请

## 六、Runtime 与对象模型

### Q18: objc_msgSend 是怎么找到 IMP 的？为什么它能做到高效？
<details>
<summary>参考答案（点击展开）</summary>

**核心三段式跳转（伪代码）：**
```c
// objc_msgSend(receiver, sel)
1. receiver 已有（消息接收者对象）
2. 获取 isa = receiver->isa
3. lookup:
   ┌────────────────────────────┐
   │ IMP cache_t::lookup(sel)  │  ← O(1) hash 查找
   │   → hit: return imp       │
   │   → miss: 搜索 method list│
   │       → 找到: cache + return
   │       → 找不到: 消息转发   │
   └────────────────────────────┘
```

**缓存查找（cache_t::find）**：
```objc
static inline IMP cache_t::find(SEL s) {
    // 用 sel 的指针地址做 hash
    // bucket_t *buckets = 动态数组
    // 索引 = (s >> 2) & mask  // 伪代码，实际是多种 hash
    // 线性探测或掩码碰撞处理
}
```

**为什么高效**：
- **方法缓存是 lazy 的**：第一次调用时缓存 nil，调用链走到 class method list 才缓存
- **Hash 查找 O(1)**：不遍历整个 method list
- **缓存亲和性**：大多数方法会重复调用，同一个 sel 多次调用直接命中缓存

**消息转发链（method miss 时）**：
```
objc_msgSend
  → class_resolveMethod
    → _objc_msgForward_impcache  // 缓存未命中
  → forwarding
    → objc_object::forwarding
      → methodSignatureForSelector / resolveInstanceMethod
        → doesNotRecognizeSelector  // 抛出异常
```

### Q19: 方法缓存存在哪？cache_t 的结构是什么？rehash 发生在什么时候？
<details>
<summary>参考答案（点击展开）</summary>

**方法缓存存在对象头（isa）的 extra 区域或 Side Table 中**，但实际上：
- **类对象**（Class）：每个类对象有一个 `cache_t`，存储继承和重写的方法缓存
- **Meta-Class**：同样有自己的 cache

**cache_t 结构（arm64）：**
```objc
struct cache_t {
    struct bucket_t *_buckets;   // hash 表的桶数组
    mask_t _mask;                 // _buckets 的大小 - 1（用于 & mask 取模）
    mask_t _occupied;            // 已占用的桶数量
};

struct bucket_t {
    cache_key_t _key;   // SEL 指针作为 key（存储的是 sel 地址）
    IMP _imp;          // 方法实现
};
```

**Hash 算法**：
```c
// 插入
index = (sel >> 2) & mask;  // & mask 比 % mask 快（位运算）
// 碰撞：线性探测 next = (index + 1) & mask
```

**扩容（rehash）时机**：
- 当 `_occupied > _mask * 3/4` 时触发
- 新容量 = 旧容量 * 2（向上取整到 2 的幂）
- 重新 hash 所有已缓存的 bucket

**为什么用 3/4 作为阈值？** 平衡空间利用率（3/4 意味着 25% 空闲槽）和碰撞概率，保证 hash 性能。

**注意**：Objective-C 的 cache 是**类级别共享**的（而不是实例级别），同一个类的所有实例共享同一个 cache 副本。
</details>

### Q20: Category 和 Extension 在内存布局上有什么区别？Category 的方法是如何"覆盖"主类方法的？
<details>
<summary>参考答案（点击展开）</summary>

**Extension（类扩展）**：
- 编译时合并到主类，**不是独立文件**
- 没有自己的内存布局
- 添加的方法、属性、实例变量直接写入主类结构

**Category**：
- 编译后生成独立的 `category_t` 结构体
- 运行时通过 `objc运行时` 的 `attachCategories` 合并到主类

**Category 合并流程（Runtime）**：
```
load_images
  → realizeClass // 第一次加载类时
  → attachCategories(class, cats, /*not metaclasses*/)
    → 对每个 category：
        → 将 category 的 method_list、property_list、protocol_list
           添加到主类的对应列表**头部**（不是覆盖！）
    → 主类 method_list 现在是：[Cat_m1, Cat_m2, ..., Original_m1, ...]
```

**"覆盖"的真相**：
- **Category 方法不会真正覆盖主类方法**
- 只是因为 method lookup 时**从列表头部开始搜索**，先匹配到 Category 的实现
- 主类的实现仍然存在，只是排在了后面
- 如果 Category A 和 Category B 有同名方法，**谁后编译谁在前**（编译顺序决定）

**重要特性**：
```objc
// Category 可以添加 property（只生成 getter/setter 声明，不生成 ivar）
// Category 可以添加协议（@protocol）
// Category 无法添加 ivar（实例变量）
// Category 无法添加成员变量（因为 ivar 存储在固定偏移量）
```

**为什么 Category 不能添加 ivar？** 因为 ivar 的存储位置在对象内存布局中是固定的（编译时确定偏移量），而 Category 是运行时附加的，无法改变已有对象的内存布局。
</details>

## 七、进阶：MMKV / Tagged Pointer / 特殊场景

### Q21: 什么是 Tagged Pointer？NSString、NSNumber 在什么条件下使用它？它为什么能"省内存"又"提速"？
<details>
<summary>参考答案（点击展开）</summary>

**Tagged Pointer 定义**：
- 在 64 位架构下，如果一个对象的地址空间**不需要全部 64 位**，苹果用最后 1 位（或几位）作为**类型标记**，将数据直接存储在指针本身中
- **不再需要分配堆内存**，指针本身就是数据

**NSString 何时启用 Tagged Pointer**：
```objc
// 启用条件（iOS 9+）：
// 1. 字符串长度 ≤ 9 个字符（具体依系统版本）
// 2. 内容是纯 ASCII（无特殊 Unicode 编码）
// 3. runtime 检测到满足条件，直接用指针存储

// 指针结构（伪代码）：
// [1 bit = 1 (tagged)] [4 bit tag type] [59 bit payload]
```

**NSNumber 何时启用**：
```objc
// 能用 63 位整数表示的值，全部用 Tagged Pointer 存储
// 例如：NSNumber @42 → Tagged Pointer
//       NSNumber @(1<<60) → 堆对象（Tagged Pointer 存不下）
```

**省内存**：不需要 malloc 分配 + isa + 引用计数，指针本身就是对象

**提速**：
- 分配：`Tagged Pointer` 无需任何分配，直接从寄存器取
- 释放：不需要 release/dealloc
- 访问：不需要解引用 `*ptr`，直接 decode 指针值
- 缓存：runtime 有全局 `gStorageStrings` 小字符串缓存池，相同内容直接返回同一地址

** Tagged Pointer 的坑**：
```objc
// 你以为的：
NSString *s = @"hello";  // 直接是 char* C 字符串，不是对象！

// 真正的：
// Tagged Pointer 字符串不等于 __NSCFConstantString
// Tagged Pointer 字符串是 runtime 动态创建的
// 用 %p 打印会看到低位有 tag bits
// 用 object_getClass 会返回 nil（Tagged Pointer 没有 isa）
```
</details>

### Q22: 如果让你设计一个自定义内存分配器，要求对小对象（<256B）做对象池管理，你会用什么数据结构？为什么？
<details>
<summary>参考答案（点击展开）</summary>

**推荐数据结构**：**Free List（单向/双向链表）+ Size Class 分桶**

```
Pool[16B]  → [obj1] ↔ [obj5] ↔ [obj9] ...
Pool[32B]  → [obj2] ↔ [obj6] ...
Pool[64B]  → [obj3] ↔ [obj7] ...
Pool[128B] → [obj4] ↔ [obj8] ...
```

**为什么不用 bitmap？**
- 小对象对象数量极大，bitmap 开销可以接受，但遍历效率不如链表
- Free List 分配/释放：O(1)（头插/头删）
- 如果需要精确大小匹配，bitmap 需要逐位扫描找空槽

**为什么不用 slab 分配？**
- slab 适合固定大小对象池（如内核对象分配）
- 如果需要支持多种 size class（16/32/48/64/...），slab 会有内部碎片

**组合最佳方案**：
```c
struct PoolArena {
    FreeList *small_pools[NUM_SIZE_CLASSES]; // size class 分桶
    Bitmap page_bitmap;  // 追踪 page 内的槽位占用
    void *memory_page;   // 预分配的 4KB 大页
};
```

**分配流程**：
```
allocate(size):
    size_class = round_up_to_size_class(size) // 找最近的 class
    bucket = &arena.small_pools[size_class]
    if (!bucket->isEmpty()) {
        return bucket->popHead(); // O(1) 弹出
    }
    // bucket 空，从 page 或 mmap 补充新 slab
```

**释放流程**：
```
free(ptr, size):
    size_class = round_up_to_size_class(size)
    arena.small_pools[size_class].pushHead(ptr) // O(1) 插入
    // 无需归还给操作系统，保持 warm
```

**为什么 O(1) 很重要？** 因为内存分配器会被高频调用（每次 alloc/dealloc），O(1) 保证不会成为性能瓶颈。苹果的 Zone malloc 也是基于类似的 Freelist 思想。
</details>

### Q23: NSNotificationCenter 默认是 strong 引用观察者，为什么通常不会造成泄漏，但 removeObserver 忘记调用却会？
<details>
<summary>参考答案（点击展开）</summary>

**为什么不泄漏（正常情况）？**
- `addObserver` 注册后，**观察者被 NSNotificationCenter retain**
- 但观察者**不 retain NotificationCenter**（它是单例 `defaultCenter`，全局唯一）
- 正常情况下，观察者应该在 `dealloc` 或 `viewWillDisappear` 前主动调用 `removeObserver`
- 如果正常移除 → 引用关系干净

**为什么会泄漏（忘记 removeObserver）？**
```
VC.viewDidLoad
  → [NSNotificationCenter.defaultCenter addObserver:self
                                            selector:@selector(handle)
                                                name:@"Event"
                                              object:nil];
  → defaultCenter.retain_count↑ （持有 self）

VC 被 pop / dismiss
  → VC.dealloc 预期执行
  → 但 dealloc 中未 removeObserver
  → removeObserver 内部会：self 还在 center 的观察者列表中
  → dealloc 尝试向已释放的对象发消息
  → Crash 或 内存泄漏（center 持有 dangling pointer）
```

**关键点**：
- **center 持有 observer 是单向的**，observer 本身不持有 center
- `defaultCenter` 是单例，**生命周期 ≈ 进程生命周期**
- 忘记 `removeObserver` 的后果：**center 的 observers 列表中残留无效指针**
- **在 iOS 9 之前**：会导致**通知发送时向野指针发消息 → EXC_BAD_ACCESS**
- **在 iOS 9+**：runtime 会检测 observer 是否已 dealloc，如果已 dealloc 则自动移除（但会产生日志警告）

**最佳实践**：
```objc
// 在 dealloc 中（注意不能调用 self 的方法）
- (void)dealloc {
    [[NSNotificationCenter defaultCenter] removeObserver:self];
    // 或者用 __weak 方式提前移除
}

// 更安全的写法（使用 Block 观察者，返回一个 token）
id token = [[NSNotificationCenter defaultCenter]
    addObserverForName:@"Event"
                object:nil
                 queue:nil
            usingBlock:^(NSNotification *note) {
                // Block 会 retain observer，这里也是潜在的 cycle！
            }];
// 用完后：[center removeObserverWithToken:token]; // iOS 9+
```
